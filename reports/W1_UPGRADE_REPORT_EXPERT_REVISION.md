## 1. 必须纠正的根本问题（按严重度排序）

### 🔴 问题一：把"高命中率"误当成"优势/edge"——这是会让系统自我欺骗的核心陷阱

原报告假设：校准模型概率 → 选最高的 → 就得到一个高命中率推荐。但有一个**绕不过去的事实**（我们前几周已实证）：

> **W1 的 λ / score matrix 是从市场赔率反解出来的**（devig 1X2 + OU → λ）。所以模型对 "Under 2.75" 的概率，本质就是**市场隐含概率的再表达**。

后果：
- 对它做历史校准，会把它拉到 ≈ **市场收盘概率**。
- 于是 Opportunity Selector 选出的 TOP_PICK，几乎永远是**市场本来就定价为高概率的方向**（大热、明显的大/小球）。
- 这些方向**命中率当然高**（比如 65%）——但那是**因为市场早就这么定价**，不是你比市场更准。
- `TOP_PICK conditional hit rate` 指标会很漂亮，但它度量的是**"你有没有挑热门"**，不是**"你有没有跑赢市场"**。

**这是这类系统最危险的地方：推荐样本命中率很高 → 看起来成功 → 实际零优势。** 必须在报告里写死这一点，否则会拿"62% 命中率"骗自己和总监。

**修订**：
- 明确定义当前系统的 TOP_PICK = **"市场共识里校准后置信度最高的方向"**，并在每条推荐上**诚实标注"≈市场共识，非独立优势"**。
- 把 `probability_gap_vs_market`（模型 vs 市场隐含概率之差）**升为唯一的真 edge 信号**——但同时承认：**因为模型=市场，这个 gap 现在 ≈ 0**（就是我们一直说的 market_fit_error≈0）。
- **真正的 edge 只能来自独立于市场的因子-λ 模型**（见问题三），在它被 walk-forward 证明之前，系统不得宣称有优势。

### 🔴 问题二：原报告的目标命名/文案与项目红线正面冲突

原报告通篇用 **"最高命中率机会选择系统"、"入场价值"、"升盘"、"值得输出的机会"**——这是**投注/推介(picks)产品**的语言。

但 W1 一直的红线（你自己反复定的）：**不是投注平台、不输出资金建议、不承诺命中率、不把模型-市场分歧表述为投注机会**。

直接采用原报告的框架=**改变项目的合规定位**。这必须你**显式拍板**：

- **方案 A（推荐，守住红线）**：把目标重命名为 **"校准后置信度排序 + 单点研究结论 / 跳过"**。系统选出"本场最值得说的一个分析读数（或跳过）"，用**校准质量 + 区间命中 + 覆盖率**评估，**不**用"命中率最大化"作为业务目标，**不**出现"入场/升盘/机会"等投注词。SKIP/WAIT/BLOCKED 保留。
- **方案 B（改红线）**：如果你确实要做"命中率最高的推介"产品，那是一次**有意识的红线变更**，需要你和总监正式签字，并承担相应的合规取舍。我作为专家**建议 A**。

> 命名建议：`W1 Opportunity Selector` → **`W1 Single-Read Selector`（单点结论选择器）** 或 **`W1 Confidence Ranker`**；输出 `W1_TOP_PICK` → **`W1_PRIMARY_READ`**（主结论），语义是"研究主结论"而非"推介"。

### 🟠 问题三：λ 的因子调整（9.4–9.8）目前不可建——它就是"独立因子模型"，data-gated

原报告 9.4–9.8 要用**首发/伤停/天气/裁判/战术**去调 `λ_home`/`λ_away`。但：
- W1 **当前不从这些因子调 λ**（λ 来自赔率）。
- 真要做=**独立因子-λ 模型**，需要 xG/射门/PPDA/阵容强度/裁判库等数据——**多数本地没有、且红线是不抓取**。

**修订**：把 9.4–9.8 整段标为 **`W1_FACTOR_LAMBDA_MODEL` 研究轨**（已登记），明确：
- 先 `W1_FACTOR_LAMBDA_DATA_INVENTORY_V1` 盘点数据可得性，**拿不到不硬做**；
- 任何 λ 因子调整**必须 walk-forward 证明跑赢市场后**才接生产；**绝不直接上线、不改 `DEFAULT_RHO`、不替换 score engine**（红线）。
- 在它落地前，opportunity_score 里的 lineup/injury/weather 扣分**只能是"不确定性/风控降级"，不能宣称提升了预测准确度**。

### 🟠 问题四：per-bucket 校准需要的数据量，当前远不够

原报告 8.x 的校准分桶（OU线 × 总进球 × 集中度 …）会产生大量桶，每桶要**上千场**才稳。W1 只有 128 场 FULL 回测 + 正在累积的 prospective ledger。

**修订**：
- 初期**只做全局 shrinkage 到联赛/市场先验**（原报告阶段三已提到，正确），**不建庞大 per-bucket 注册表**。
- 诚实承认：**短期内 p_calibrated ≈ 市场**；校准当前的价值是**诚实(防过度自信)**，不是 edge。
- 校准注册表随 ledger 样本增长**逐步、分桶上线**，每个桶要有最低样本量门槛，否则回退到先验。

## 2. 修订后的系统定位（替换原报告第三章）

> W1 **不是**"最高命中率推介系统"。W1 是 **"校准后单点研究结论选择器"**：在 OU/AH/BTTS/1X2/比分池中，选出**本场最值得表达的一个分析读数**（或 `WAIT`/`SKIP`/`BLOCKED`），并**诚实标注它相对市场是否有独立优势**。
>
> 评估指标：**校准质量（Brier/LogLoss/ECE）+ 区间/方向命中 + 覆盖率 + `probability_gap_vs_market` 的真实性**。
> **不承诺命中率**，**不构成投注/入场建议**，**不把模型-市场分歧表述为投注机会**。

## 3. 修订后的分阶段落地（验证/数据 gated，不接 API、不动引擎）

| 阶段 | 内容 | 红线/前提 | 可现在做? |
|---|---|---|---|
| **A. Candidate 统一 + 视图分离（只读派生）** | 把 OU/AH/BTTS/1X2/ScorePool 统一成 Candidate（**全部从现有 score matrix 只读派生**）；Director/Analyst 双视图（≈ 已做的 declutter）；ledger 加 candidate 字段 | 不改引擎/不接 API；输出标注"≈市场共识，非独立优势" | ✅ 可做（纯展示+只读派生） |
| **B. 校准研究（global shrinkage 起步）** | 单市场校准，先全局收缩到先验；用 128 回测 + ledger 验证；产研究报告 | 研究/prototype，不接生产决策；不宣称 edge | ✅ 研究可做（数据有限，结论先弱） |
| **C. 独立因子-λ（唯一真 edge 来源）** | `DATA_INVENTORY` → 可验证子集建模 → walk-forward vs 市场 | **跑赢市场前不动引擎/`DEFAULT_RHO`**；不接 API（你说暂不接） | ⏳ 先盘点数据，无数据不硬做 |
| **D. opportunity_score / TOP_PICK 选择器** | 风险扣分 + 单点选择 + 阈值 | 阈值必须 ledger 验证；文案守红线（PRIMARY_READ，非推介） | ⏳ 待 A/B 稳定后，**且**红线问题确认后 |

**关键顺序**：A（现在可做、低风险）→ B（研究）→ C（独立 edge，最有价值但 data-gated）→ D（选择器，最后且需红线确认）。**不要一上来就建全套 calibration registry + risk penalty + selector**——在没有数据和独立 edge 之前，那只会是一套"跟着市场热门走、再扣点分"的精致包装。

## 5. 一句话结论

原报告的**产品直觉对**（单点 + 跳过 + OU/AH 优先 + 校准 + ledger + 视图分离），但它**把"挑市场热门"误当成"系统有优势"**，并用了**越界的推介语言**。修订后：**守住"研究/概率/风控"定位**，**先做只读的 Candidate 统一 + 视图分离**，**把"真优势"诚实地交给 data-gated 的因子-λ 研究**——在它被证明之前，系统老老实实说"我现在≈市场，没有独立优势"。
